# RetellAI Full API Parity — Program Roadmap

**Date:** 2026-06-24 · **Type:** Program decomposition artifact (not a single implementable spec) · **Status:** Kickoff — awaiting review

> Review this before we drill into Phase 1. Each phase becomes its own `spec → plan → implementation` cycle and its own squash-merged PR.

## Brainstorm decisions captured

This roadmap is the product of a brainstorming session that locked four keystone decisions:

1. **Goal = full RetellAI API parity** (north star) so any RetellAI client repoints its base URL to us with zero code changes.
2. **No specific client yet → generic readiness** against the published contract (not one CRM's traffic).
3. **Roadmap-first**, then drill into Phase 1.
4. **Correctness oracle = pinned RetellAI OpenAPI snapshot + pinned official SDK + per-endpoint conformance tests.** "Frozen" = conformance-green against the pinned snapshot.

The endpoint inventory below was generated by parsing RetellAI's real `openapi-final.yaml` (84 operations, v3.0.0) and their official Python/TypeScript SDKs, cross-mapped against our existing `apps/api/src/usan_api/compat` surface. The 501 path-drift / phantom-endpoint claims in §2.3 and §3 were verified directly against `compat/routers/unsupported.py`.

## 1. Goal & Non-Goals

**Goal.** Achieve full RetellAI API parity so that **any** existing RetellAI client (using the official SDKs or raw HTTP) can repoint its base URL from `https://api.retellai.com` to our service and operate with **zero code changes**. Parity means: identical paths (including per-endpoint version segments), identical request/response/webhook JSON shapes, identical auth scheme, identical error envelope, identical pagination semantics.

**Non-Goals.**
- **Bug-for-bug undocumented-quirk parity.** We match the *documented + SDK-generated* contract (the pinned OpenAPI snapshot + pinned SDK), not RetellAI's internal undocumented behaviors, latency characteristics, or unspecified edge-case responses.
- **No specific client tuning yet.** No client is onboarded, so we build to **generic readiness** against the oracle, not to one CRM's observed traffic. (The 21 `PENDING-FREEZE` markers that need a *captured* oracle are resolved against the OpenAPI YAML field schemas, not a single client capture.)
- **Capability we cannot back.** Where our engine genuinely cannot perform a behavior, we return a **documented 501**, not a fake 200 (see §2).

## 2. Foundational Principles

These apply to **every** phase. A phase is not "done" until its endpoints satisfy all of these.

### 2.1 The Pinned-SDK Conformance Oracle (foundational)
- **Pinned OpenAPI snapshot (authoritative for *which routes exist* and *field schemas*):**
  `https://docs.retellai.com/openapi-final.yaml` — `info.title` "Retell SDK", `info.version` 3.0.0, `openapi: 3.0.3`, server `https://api.retellai.com`. **84 path+method operations.** Also pin the Stainless spec hash the SDKs are generated from: `openapi_spec_hash = 18ce2ce5e3ecffc21a95f1bc21c1c82c` (`configured_endpoints = 78`). Vendor a frozen copy of the YAML into the repo (e.g. `apps/api/tests/compat/oracle/openapi-final.3.0.0.yaml`) so CI is hermetic.
- **Pinned official SDKs (authoritative for *method/type-name mapping* and the `verify()` helper):**
  - **Python:** PyPI **`retell-sdk == 5.53.0`** (`from retell import Retell`; requires-python ≥3.9). **Primary SDK pin** for the conformance harness (ships `retell.verify(...)`; the Python `api.md` is the resource map of record).
  - **TypeScript:** npm **`retell-sdk == 5.40.0`** (`import Retell from 'retell-sdk'`). Secondary cross-check (TS lags the Python patch stream; resource/endpoint surface is byte-identical between them).
  - Both are Stainless-generated from the same spec snapshot → identical resource surface across languages.
- **Conformance test per compat endpoint.** Every compat endpoint gets an automated conformance test asserting its request acceptance, response body, and (where applicable) webhook payload **validate against the pinned RetellAI component schemas** (decode SDK response models / validate against the frozen YAML's `*Response`/`*Params` schemas).
- **"FROZEN" definition.** An endpoint (or phase) is **FROZEN** when its conformance tests pass green against the pinned snapshot and a contract test locks the byte-shape so future changes are caught as diffs. Re-pinning the SDK/spec is a deliberate, reviewed bump (Stainless regenerates near-daily).

### 2.2 Capability-Bounded Parity
API compat **cannot exceed engine capability.** A response field we cannot truthfully populate is emitted per the oracle's nullability rules, never faked. Where the engine cannot perform the *action* at all, see §2.3.

### 2.3 Documented-501 Policy (not-yet-built)
Any oracle endpoint we have not yet implemented returns **HTTP 501** with the RetellAI error envelope (`{"status":"error","message":"not_supported:<endpoint>"}`) at its **exact versioned path**, and is documented in the compat OpenAPI. **A 501 at the wrong path is parity-equivalent to a 404** — the 501 router MUST be generated from the pinned YAML, not hand-named. (Verified current debt in `compat/routers/unsupported.py`: `/list-chat` vs `/v3/list-chats`, `/add-voice` vs `/add-community-voice`, `/search-voice` vs `/search-community-voice`, `/create-test-case` vs `/create-test-case-definition`, missing `/v2` prefixes on `/list-conversation-flows` & `/list-phone-numbers`, `/get-mcp-tools` drops `{agent_id}`; plus *phantom* stubs `/get-export-request` and `/create-test-run` that are not real oracle endpoints. Phase 1 fixes this.)

### 2.4 Per-Org Bearer Auth + RLS
`Authorization: Bearer <key>` resolves to an org via the global `compat_api_keys` table (sha256 hash + 8-char prefix lookup, constant-time compare, plaintext shown once). Every request is RLS-scoped to that org (`set_config('app.current_org', …)`, re-applied per-txn). Missing/invalid/revoked → `401`.

### 2.5 Webhook Signature
Header **`X-Retell-Signature`**, format **`v={timestamp_ms},d={hex_digest}`**, where `d = HMAC-SHA256(key=subscription_secret, msg = raw_body + str(timestamp_ms))`, lowercase hex, **no separator**. Replay window ±5 min, constant-time compare, signed over the **raw** body. CRM verifies with unmodified `Retell.verify(raw_body, secret, signature)`. (Casing of our emitted header is benign per HTTP, but a conformance assertion pins it.)

### 2.6 Error Envelope
Exact shape `{"status":"error","message":"<human string>"}` with the oracle's status-code vocabulary: **400, 401, 402, 403, 404, 409, 412, 422 ("Cannot find requested asset under given api key"), 429, 500.** Validation failures → 422 (first-field-only, no PHI). Catch-all → 500 (type-name only).

### 2.7 Rate Limiting
Dedicated elevated compat bucket `COMPAT_KEY_RATE_LIMIT` (default `600/minute`), keyed per bearer key, disjoint from operator/auth buckets. Over-limit → `429` in the RetellAI envelope.

### 2.8 Idempotency
Mutating create endpoints must define an idempotency posture (dedupe window for double-fire on `create-phone-call`/`create-batch-call`; resolve the existing `call_create.py` `PENDING-FREEZE` on the double-fire window). Re-submission within the window must not double-dial.

## 4. Phase Decomposition

Each phase = its own **spec → plan → implementation** cycle and its own **squash-merged PR** (per repo workflow). Merged ≠ deployed (gated by a `v*` tag). Magnitudes: **S** ≤ a few days, **M** ~1 week, **L** ~2–3 weeks, **XL** ~1 month+.

### Phase 1 — Activate, validate-against-SDK, freeze & harden the core calling surface
**Scope (endpoints):** Everything currently ✅/🟡 plus the Pri-1 silent-404 fixes: `register-phone-call`, `delete-call`, `update-live-call`, `delete-agent-version`, `POST /v2/list-agents`, bare `publish-agent`; complete `stop-call` (force-hang connected rooms), `update-live-call` mid-call vars, broader `/v3/list-calls` `filter_criteria`, `?version` honoring on `get-agent`/`get-retell-llm`. **Regenerate the 501 router from the pinned YAML** so all ⛔/❌ paths return 501 at their exact versioned paths (fix `/list-chat`→`/v3/list-chats`, `/add-voice`→`/add-community-voice`, `/search-voice`→`/search-community-voice`, `/create-test-case`→`/create-test-case-definition`, prefix drift; remove phantom `/get-export-request` & `/create-test-run`). Resolve all **21 `PENDING-FREEZE`** markers against the YAML field schemas. Ship the **admin-UI compat-key management** UI (issuance/list/revoke over the existing `/v1/admin/compat-keys`). Flip `COMPAT_*` on and issue a key in a controlled env.
**Capability needed:** Already present (LiveKit SIP, dialer, Postgres, GCS, webhook poller). New plumbing: LiveKit room force-hangup; agent-RPC mid-call var injection.
**Dependencies:** None (this is the base). Requires the conformance harness (workstream §5) to land first within this phase.
**Magnitude:** **L** (small per-endpoint, large in count + the freeze + admin-UI).
**Exit criteria:** All core-calling endpoints implemented at exact versioned paths; every ⛔/❌ oracle path returns a **correct-path 501**; admin-UI can issue/list/revoke compat keys; webhook signature + envelope + rate-limit + auth conformance assertions pass; **all 21 PENDING-FREEZE resolved; conformance tests green + contract FROZEN** for the entire phone-call / agent / retell-llm / batch / voices / concurrency / outbound-webhook surface; a real RetellAI Python SDK (5.53.0) smoke test places + retrieves + lists + stops a call end-to-end against our service.

### Phase 2 — Phone Numbers + Analytics/Exports
**Scope:** `create/import/get/update/delete-phone-number`, `GET /v2/list-phone-numbers`; `GET /v2/list-export-requests` (+ export job → GCS signed URL).
**Capability:** Telnyx Numbers API (search/order/release) + LiveKit trunk binding per-org; bulk export job to GCS.
**Dependencies:** Phase 1 (auth/RLS/key plane). Product decision on numbers posture (see §7).
**Magnitude:** **M** (numbers) + **S** (exports).
**Exit criteria:** Per-org Telnyx number lifecycle works behind the compat shapes; export request lifecycle returns oracle-shaped artifacts; **conformance green + frozen** for both areas; if numbers stay deliberately unsupported, the 501s are documented and that posture is recorded as a contract decision.

### Phase 3 — Web Calls (LiveKit WebRTC)
**Scope:** `POST /v2/create-web-call` (returns `access_token` + `call_id`); web calls flow through the shared call CRUD/lifecycle (`get-call`, `update-call`, `stop-call`, `delete-call`, `list-calls`, webhooks).
**Capability:** LiveKit WebRTC native `access_token` issuance — the single closest primitive match in the stack (RetellAI's web-call token model *is* LiveKit's).
**Dependencies:** Phase 1 (shared call lifecycle + webhook emit must be frozen first).
**Magnitude:** **M**.
**Exit criteria:** A browser using RetellAI's web SDK connects via our `access_token`; web-call objects serialize identically to phone-call objects through the shared endpoints; `call_started/ended/analyzed` webhooks fire for web calls; **conformance green + frozen** for `create-web-call` and web-call serialization.

### Phase 4 — Chat & SMS Chat
**Scope:** Chat sessions (`create-chat`, `get-chat`, `update-chat`, `end-chat`, `delete-chat`, `POST /v3/list-chats`, `create-chat-completion`), `create-sms-chat`, `rerun-chat-analysis`; **Chat Agents** full CRUD + versioning (`create/get/update/list/delete-chat-agent`, `get-chat-agent-versions`, `publish-chat-agent`, shared `create/delete-agent-version`); chat webhook events.
**Capability:** Vertex (Gemini) for chat turns; **Telnyx Messaging** for SMS (gated on `send_sms` provisioning, already flagged INERT in memory); pgvector/Postgres for session + message history.
**Dependencies:** Phase 1 (agent/version model + webhook plane). Telnyx Messaging provisioning.
**Magnitude:** **L**.
**Exit criteria:** Text + SMS chat sessions run turn-by-turn; chat agents version/publish; `chat_ended`/`chat_analyzed` (and confirmed `chat_started`) webhooks fire signed; `/v3/list-chats` keyset pagination matches calls; **conformance green + frozen** for the chat + chat-agent + SMS surface.

### Phase 5 — Knowledge Base (pgvector RAG)
**Scope:** `create/get/list/delete-knowledge-base`, `add-knowledge-base-sources` (multipart files/urls/text), `delete-knowledge-base-source/{kb}/source/{src}`; bind KBs to agents for retrieval at call/chat time.
**Capability:** **pgvector** embeddings (strong primitive fit) + Vertex embedding+generation + a fetch/chunk/ingest pipeline (urls/files/text) with source-status tracking; GCS for uploaded files.
**Dependencies:** Phases 1 & 4 (agents/chat consume KB at runtime).
**Magnitude:** **L** (the ingest/chunking pipeline + async source-status is the work; the vector store is owned).
**Exit criteria:** KB create + source add/delete with correct async status shapes; retrieval demonstrably affects agent answers; `delete_source` two-segment URL + `(source_id, *, knowledge_base_id)` mapping honored; **conformance green + frozen** for the KB surface.

### Phase 6 — Conversation Flow (DAG runtime) + Components
**Scope:** `create/get/update/list/delete-conversation-flow` and the 5 `conversation-flow-component` endpoints; bind a conversation-flow response engine to agents.
**Capability:** **App-layer DAG runtime (greenfield)** — node/edge graph execution semantics over Vertex; reusable sub-flow Components model. No vendor provides Retell's flow semantics.
**Dependencies:** Phases 1 & 5 (agents bind a response engine; flows reference KBs/tools).
**Magnitude:** **XL**.
**Exit criteria:** A conversation-flow agent drives a real call/chat through a multi-node graph with components; flow + component CRUD round-trips at oracle shapes; **conformance green + frozen** for both areas.

### Phase 7 — Long tail: Test Suites/Simulation, Playground, MCP, Voice Clone
**Scope:** Test Suites (10 endpoints — note `update-test-case-definition` is **PUT**), `agent-playground-completion`, `get-mcp-tools/{agent_id}`, `clone-voice` + `add/search-community-voice`.
**Capability:** Greenfield-heavy — **simulation harness** (persona LLM + success-criteria grader + run orchestration over our own call engine); **MCP tool registry/resolver**; **Cartesia voice-clone** wiring + community-library concept; Playground is a thin Vertex single-turn wrapper (do this early/cheaply within the phase).
**Dependencies:** All prior phases (sim/playground exercise agents, flows, KBs, web/chat calls).
**Magnitude:** **XL** (Test Suites dominate) — Playground/Voice Clone are **S–M** sub-items.
**Exit criteria:** Each sub-area implemented or, where genuinely unbackable, documented-501; sim harness runs a batch test producing oracle-shaped runs; **conformance green + frozen** for every remaining oracle endpoint, so the **entire 84-operation surface is FROZEN or documented-501**.

## 6. Sequencing & Dependencies

1. **Phase 1 is the keystone** — it activates the inert surface, builds the conformance harness everything else depends on, fixes the silent-404s (correct-path 501s), ships key-management UX, and **freezes** the high-value calling surface. No later phase can claim parity without the harness.
2. **Phases 2 & 3 are independent of each other** and can run in either order after Phase 1 (numbers/exports are isolated; web calls reuse the now-frozen shared call lifecycle). Web Calls (Phase 3) is the **highest parity-per-effort** real-capability win (LiveKit primitive already issues tokens) — schedule it early if a client needs web.
3. **Phase 4 (Chat)** needs the frozen agent/version + webhook plane from Phase 1 and Telnyx Messaging provisioning.
4. **Phase 5 (KB)** feeds runtime retrieval, so it precedes **Phase 6 (Conversation Flow)**, which binds KBs/tools into a graph and is the heaviest build.
5. **Phase 7 (long tail)** is last because Test Suites/Simulation and MCP are pure greenfield that exercise *all* prior capabilities; Playground/Voice Clone are cheap and can be pulled forward opportunistically if a phase finishes early.

Ordering rationale: **cheapest-parity-first** (eliminate 404s, freeze what we already have) → **best-primitive-fit next** (LiveKit web, Telnyx numbers/SMS, pgvector KB) → **greenfield last** (DAG runtime, simulation, MCP).

## 7. Risks & Open Questions

- **Build-against-SDK blind spots.** Six spec-only operations don't exist as SDK methods. Building from `api.md` alone silently omits them. **Mitigation:** the YAML (84 ops) is the route oracle; the matrix in §3 already enumerates them.
- **Per-endpoint versioning trap.** Mixed `/v2`, `/v3`, and un-prefixed paths on one host. A single wrong version segment breaks a client. **Mitigation:** 501 router + real routes generated from the YAML; conformance asserts exact paths.
- **PENDING-FREEZE field shapes (21 markers).** Need exact request/response field schemas from the YAML, not type names: `override_agent_version` int-vs-string, `telephony_identifier`/SIP-INVITE mapping, per-call sentiment nullability, recording/transcript shape, disconnection_reason vocabulary, voice alias strings (`retell-`/`11labs-`). **Open:** these resolve from the YAML now (no client capture), but a real client capture later may surface undocumented expectations we explicitly do **not** chase (non-goal).
- **Chat webhook enum uncertainty.** `chat_started` appears in prose but not the machine enum (`chat_ended`/`chat_analyzed`). **Open:** pin from the raw YAML before claiming chat-webhook parity (Phase 4).
- **SDK drift.** Stainless regenerates near-daily; TS (5.40.0) and Python (5.53.0) are on diverging patch streams. **Mitigation:** pin both; re-pin is a reviewed bump with a fresh conformance run.
- **Phone-Numbers posture.** Real per-org Telnyx provisioning vs deliberate 501 is a **product/tenancy/billing decision**, not a tech gap. **Open question for Phase 2.**
- **PHI egress via webhooks/KB ingest.** Outbound webhooks and KB url/file ingest move PHI off-box. **Mitigation:** `COMPAT_WEBHOOK_ALLOWED_HOSTS` allow-list + SSRF guard (already built); Vertex (BAA-covered) for embeddings/generation; KB ingest must respect the same egress discipline.
- **Greenfield magnitude.** Test Suites/Simulation, Conversation-Flow runtime, and MCP have **no vendor primitive** and are the largest, least-certain builds — kept last to de-risk the parity surface that real clients depend on first.
- **Idempotency window.** `create-phone-call`/`create-batch-call` double-fire dedupe window is unresolved (`call_create.py` PENDING-FREEZE) — must be pinned in Phase 1 to avoid double-dialing on client retries.
