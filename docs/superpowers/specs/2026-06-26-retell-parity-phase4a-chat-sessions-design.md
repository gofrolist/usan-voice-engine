## 1. Context & Scope

"Phase 4: Chat + SMS" is **16 oracle operations across three independent subsystems**. Per the
brainstorming scope rule it is decomposed into three sub-phases, each its own PR (mirroring the
1a/1b/1c cadence):

- **4a (this spec)** — `api_chat` chat sessions, **live**. 7 ops.
- **4b (later)** — SMS chat (`create-sms-chat` + inbound Telnyx Messaging webhook; async two-way). 1+ ops.
- **4c (later)** — chat-agent management CRUD (7 ops) + `rerun-chat-analysis`.

### 4a serves these 7 ops (today all 501 in `compat/routers/unsupported.py:46–52`)

| # | Method | Path | Success | Response |
|---|---|---|---|---|
| 1 | POST | `/create-chat` | 201 | `ChatResponse` |
| 2 | POST | `/create-chat-completion` | 201 | inline `{messages: array<MessageOrToolCall>}` |
| 3 | GET | `/get-chat/{chat_id}` | 200 | `ChatResponse` |
| 4 | POST | `/v3/list-chats` | 200 | `allOf[PaginatedResponseBase, {items: array<V3ChatResponse>, total?}]` |
| 5 | PATCH | `/update-chat/{chat_id}` | 200 | `ChatResponse` |
| 6 | PATCH | `/end-chat/{chat_id}` | 204 | (no content) |
| 7 | DELETE | `/delete-chat/{chat_id}` | 204 | (no content) |

### Stays 501 after 4a (surface-coverage gate stays green; entries remain in `_UNSUPPORTED`)

`create-sms-chat` (→ 4b); `rerun-chat-analysis` + the 7 `*-chat-agent` ops (→ 4c).

### The four locked design decisions

1. **Scope:** 4a (`api_chat` sessions) only.
2. **Completion posture:** **live via Vertex** — `create-chat-completion` calls the LLM and returns a real agent reply.
3. **Data model:** **dedicated `chat_sessions` + `chat_messages` tables** (migration `0042`); chat rows never touch `calls`.
4. **Tools:** **text-only** — agent replies are `role=agent` text; no tool execution, no tool-call/transition message roles in 4a.

## 2. Why "live" is feasible without touching the voice worker

`apps/api` already owns a **direct, synchronous Vertex AI text path** used by the admin profile-test
endpoint (`routers/admin_profile_tests.py`) — no LiveKit, no `services/agent` import (the two trees must
not import each other):

- `usan_api.vertex_test.run_vertex_turn(*, model, temperature, system_instruction, tools, contents, settings) -> VertexTurn`
  — already multi-turn (`contents: list[dict]`), runs the blocking `google-genai` SDK in `asyncio.to_thread`,
  `vertexai=True` + ADC (Vertex is BAA-covered; **never** the Gemini Developer API). Returns `VertexTurn(text, tool_calls)`.
- `usan_api.prompt_substitution.substitute(text, values)` + `build_vars(resolved, custom, *, timezone, now)`.
- `usan_api.repositories.agent_profiles.is_live_profile(db, profile_id) -> bool` — the published-profile gate.

4a reuses all three. **Text-only** means we call `run_vertex_turn(..., tools=[])` and use `turn.text`. The
exact `build_vars`/`contents` call shapes are pinned in §8.2 (mirrored from the admin-test handler).

## 3. Oracle Contract (authoritative shapes)

Source: vendored `apps/api/tests/compat/oracle/openapi-final.yaml` (v3.0.0). SDK: `retell-sdk==5.53.0`.

### 3.1 `ChatResponse` (create / get / update body)

- **Required:** `chat_id` (str), `agent_id` (str), `chat_status` (enum `ongoing|ended|error`).
- **Optional (4a populates a subset):** `version` (int), `chat_type` (enum `api_chat|sms_chat`),
  `retell_llm_dynamic_variables` (map<str,str>), `metadata` (object), `start_timestamp` (int, ms),
  `end_timestamp` (int, ms), `transcript` (str), `message_with_tool_calls` (array<MessageOrToolCall>).
- **Optional, omitted in 4a** (no post-chat analysis yet): `collected_dynamic_variables`,
  `custom_attributes` (accepted on write, see §6), `chat_cost`, `chat_analysis`.

### 3.2 `create-chat-completion` response — **inline** (no named component)

`{ "messages": array<MessageOrToolCall> }`, `messages` required. "New messages generated by the agent
during this completion … does **not** include the original input messages." In 4a every element is the
`Message` variant: `{ role: "agent"|"user", content: str, message_id: str, created_timestamp: int }`.

`MessageOrToolCall` is `oneOf[Message, ToolCallInvocationMessage, ToolCallResultMessage,
NodeTransitionMessage, StateTransitionMessage, InjectedMessage, SmsMessage]`, discriminated by `role`. A
`role=agent` text message matches exactly one variant (`Message`).

> **SDK note:** `retell-sdk` types `message_id`/`created_timestamp` as `Optional` on the generated
> `Message` model even though the oracle schema marks them required. 4a **always populates both** on every
> emitted message so the payload conforms to the strict oracle schema (and round-trips the relaxed SDK type).

### 3.3 `/v3/list-chats` response — `allOf[PaginatedResponseBase, {items, total?}]`

- `PaginatedResponseBase` = `{ pagination_key: str, has_more: bool }`.
- `items`: array<`V3ChatResponse`>. `total`: int, present only when `include_total=true`.
- **`V3ChatResponse` = `ChatResponse` with a `not` clause** forbidding the keys `transcript`,
  `message_with_tool_calls`, `scrubbed_message_with_tool_calls`. **List items MUST omit those keys
  entirely** (not `[]`, not `null` present — absent). This is conformance-critical (see §7, §8.5).

### 3.4 SDK concrete round-trip types (use the concrete type, never a Union — Phase 3 lesson)

| Op | oracle component | SDK type |
|---|---|---|
| create/get/update | `ChatResponse` | `retell.types:ChatResponse` |
| completion | (inline; items → `Message`) | `retell.types:ChatCreateChatCompletionResponse` |
| list | (inline; items → `V3ChatResponse`) | `retell.types:ChatListResponse` |

### 3.5 Request bodies

- **`create-chat`:** required `agent_id` (minLength 1); optional `agent_version` (int|str),
  `metadata` (object), `retell_llm_dynamic_variables` (map<str,str>).
- **`create-chat-completion`:** required `chat_id` (str), `content` (str). **No** dynamic-vars or metadata
  fields → the reserved-prefix guard (§6) does not apply to this endpoint.
- **`update-chat`:** all optional — `metadata`, `data_storage_setting` (`everything|basic_attributes_only`),
  `override_dynamic_variables` (map<str,str>|null), `custom_attributes` (map<str, str|number|bool>).
- **`/v3/list-chats`:** all optional — `filter_criteria` (ChatFilter), `sort_order`
  (`ascending|descending`, default `descending` by `start_timestamp`), `limit` (1–1000, default 50),
  `skip` (default 0), `pagination_key` (opaque cursor), `include_total` (bool, default false).

## 4. Data Model — dedicated tables (migration `0042`, `down_revision="0041"`)

Plain `CREATE TABLE` DDL (owner-run; no `ALTER TYPE` hazard). Both tables `TenantScoped` + `FORCE ROW LEVEL
SECURITY` with the standard per-org policy, exactly like `phone_numbers` (migration `0040`).

> **`organization_id` population (RLS, isolation-critical):** the `TenantScoped` mixin declares
> `organization_id` with a DB `server_default` of
> `COALESCE(current_setting('app.current_org', true)::uuid, default_org_id())`. The migration's column
> DDL **must** carry the same `server_default`. Repositories **MUST NOT** set `organization_id` on INSERT —
> Postgres fills it from the org-scoped compat session (`set_tenant_context` sets `app.current_org`) and it
> is read back via `RETURNING`. An explicitly-set or NULL `organization_id` would be an isolation hole.

### 4.1 `ChatStatus` native enum (`usan_api/db/base.py`)

```python
class ChatStatus(enum.Enum):
    ONGOING = "ongoing"
    ENDED = "ended"
    ERROR = "error"
```

Native PG enum `chat_status` created in migration `0042` via `postgresql.ENUM(..., create_type=False).create(...)`
with `values_callable=_enum_values` on the column (matches the `CallType` pattern from `0041`). `ERROR` is
part of the oracle enum and is **reserved** in 4a — no code path assigns it (a failed completion leaves the
session `ONGOING` for retry; see §8.2).

### 4.2 `chat_sessions`

| Column | Type | Notes |
|---|---|---|
| `id` | uuid PK | `server_default gen_random_uuid()` |
| `organization_id` | uuid | `TenantScoped` (RLS key); auto-filled (see §4 note) |
| `agent_profile_id` | uuid | FK → `agent_profiles.id` |
| `agent_version` | int | the profile's `published_version` at create time |
| `status` | `ChatStatus` | `server_default 'ongoing'` |
| `chat_type` | str | `server_default 'api_chat'`; column present for 4b's `sms_chat` |
| `dynamic_vars` | JSONB | packed: bare vars + `__meta__` (metadata JSON). `__meta_unhonored__` is a reserved key (stripped on unpack) but **unused** in 4a — no accepted-not-honored field is persisted. |
| `custom_attributes` | JSONB | `server_default '{}'` |
| `started_at` | timestamptz | set at create |
| `ended_at` | timestamptz \| null | set by `end-chat` |
| `archived_at` | timestamptz \| null | soft-delete (`delete-chat`); get/list exclude `archived_at IS NOT NULL` |
| `created_at` / `updated_at` | timestamptz | standard |

### 4.3 `chat_messages`

| Column | Type | Notes |
|---|---|---|
| `id` | uuid PK | `server_default gen_random_uuid()`; encodes to `message_id` |
| `organization_id` | uuid | `TenantScoped` (RLS key); auto-filled (see §4 note) |
| `chat_session_id` | uuid | FK → `chat_sessions.id` |
| `seq` | int | per-session ordering key (see below) |
| `role` | str | `user` \| `agent` (4a) |
| `content` | text | |
| `created_at` | timestamptz | → `created_timestamp` (ms) |

**`seq` is the authoritative ordering key** (not `created_at`, which can tie when the user and agent
messages are written in the same transaction). `seq` is a per-session integer, gap-free and strictly
increasing. It is computed as `COALESCE(MAX(seq), 0) + 1` for the session **while the parent
`chat_sessions` row is held under `SELECT … FOR UPDATE`** (acquired at the top of `create_chat_completion`,
§8.2), which serializes concurrent completions on the same session and guarantees uniqueness. A
`UNIQUE(chat_session_id, seq)` constraint backs it.

> The `calls` and `transcripts` tables are untouched. No `CallType` value is added; no admin-call-list
> or `CALLS_TOTAL` guard is needed — chat rows physically cannot leak into call-plane readers.

## 5. ID Scheme (deterministic, reversible, stored nowhere — **new** functions in `compat/ids.py`, Task 2)

Net-new, added in Task 2, mirroring the existing `_decode_hex` codec used by `encode_agent_id`/`encode_llm_id`/`encode_batch_id`:

- `encode_chat_id(uuid) -> "chat_" + hex` / `decode_chat_id(str) -> uuid` (raises `CompatError(422)` on bad prefix/hex).
- `encode_message_id(uuid) -> "message_" + hex`.

`agent_id` on chat bodies is the **existing voice** `encode_agent_id`/`decode_agent_id` (a chat uses an
existing agent profile). No reverse decode of `message_id` is needed in 4a (messages are never addressed by id).

## 6. Compat Schemas (`compat/schemas/chats.py`)

**Requests** (`ConfigDict(extra="forbid")`, reserved-prefix guard — reject any
`retell_llm_dynamic_variables`/`override_dynamic_variables` key starting with `__meta`):

- `CreateChatRequest`: `agent_id: str = Field(min_length=1)`; `agent_version: int|str|None`;
  `metadata: dict|None`; `retell_llm_dynamic_variables: dict|None`.
- `CreateChatCompletionRequest`: `chat_id: str`; `content: str`. (No vars/metadata → guard N/A.)
- `UpdateChatRequest`: `metadata: dict|None`; `data_storage_setting: Literal["everything","basic_attributes_only"]|None`;
  `override_dynamic_variables: dict|None`; `custom_attributes: dict|None`. `data_storage_setting` is
  **accepted-and-ignored** (no data-storage controls in 4a; not persisted).
- `ListChatsRequest`: mirrors `ListCallsRequest` (`limit`, `skip`, `pagination_key` with skip⊕pagination_key
  mutual-exclusion validator, `sort_order`, `include_total`, `filter_criteria`).

**Responses** (`response_model_exclude_none=True` on every route):

- `CompatChatMessage`: `role: str`, `content: str`, `message_id: str`, `created_timestamp: int`.
- `CompatChat`: `chat_id`, `agent_id`, `chat_status`, `version: int|None = None`, `chat_type: str = "api_chat"`,
  `retell_llm_dynamic_variables: dict|None = None`, `metadata: dict|None = None`, `start_timestamp: int|None`,
  `end_timestamp: int|None`, `transcript: str|None = None`, `message_with_tool_calls: list[CompatChatMessage]|None = None`.
  Every field that is empty/absent is `None` (not `{}`/`[]`) so `exclude_none` **omits** it — matching RetellAI's
  "omit null optionals" convention and, for `transcript`/`message_with_tool_calls`, satisfying the
  `V3ChatResponse` `not`-clause (§3.3) on list items. `custom_attributes`/`collected_dynamic_variables`/
  `chat_cost`/`chat_analysis` are not modeled in 4a (omitted).
- `CompatChatCompletion`: `messages: list[CompatChatMessage]`.
- `ListChatsResponse`: `{ items: list[CompatChat], pagination_key: str|None, has_more: bool, total: int|None }`
  (mirrors `ListCallsResponse`).

## 7. Serialization (`compat/chat_serializer.py`)

`serialize_chat(session, messages, *, include_transcript: bool) -> CompatChat`:

- `chat_id = ids.encode_chat_id(session.id)`; `agent_id = ids.encode_agent_id(session.agent_profile_id)`.
- `chat_status = session.status.value`; `version = session.agent_version`; `chat_type = session.chat_type`.
- `(bare_vars, metadata) = unpack_dynamic_vars(session.dynamic_vars)` (strips the reserved
  `__meta_unhonored__` key). Set `retell_llm_dynamic_variables = bare_vars or None` and
  `metadata = metadata or None` — **empty → `None` → omitted** by `exclude_none`.
- `start_timestamp = to_ms(session.started_at)`; `end_timestamp = to_ms(session.ended_at)`.
- **If `include_transcript`** (create / get / update): `transcript` = rendered text (one `"<Role>: <content>"`
  line per message, in `seq` order), `message_with_tool_calls` = `[CompatChatMessage(role=m.role,
  content=m.content, message_id=ids.encode_message_id(m.id), created_timestamp=to_ms(m.created_at)) for m in
  messages]` — **both `message_id` and `created_timestamp` always populated** (§3.2 SDK note).
  **Else** (list items): both stay `None` → omitted → `V3ChatResponse`-conformant.

`pack_dynamic_vars` / `unpack_dynamic_vars` / `to_ms` are the existing `compat/serialization.py` helpers,
reused unchanged. `pack_unhonored`/`carry_unhonored` (call-specific: `agent_override`/`current_node_id`/
`current_state`) are **not** used by chat.

## 8. Service Layer (`compat/chat_service.py`) + repository (`repositories/chats.py`)

**PHI/secret-safe & commit discipline (applies to every method below):** the compat session is **not**
autocommit — every mutation ends with an explicit `await db.commit()`. Caught exceptions are re-raised as
`CompatError(...) from None`; the **global compat exception handler** (`compat/errors.py`) logs
`type(exc).__name__` only (never message/traceback). Service functions **do not** log exception detail,
request bodies, reply text, dynamic vars, or metadata (all PHI). Audit logging (router `_audit`) carries only
org id, op name, and `chat_id`.

### 8.1 `create_chat(db, body) -> ChatSession`

1. `profile_id = ids.decode_agent_id(body.agent_id)` (422 on malformed).
2. `if not await is_live_profile(db, profile_id): raise CompatError(422, ...)`.
3. Reserved-prefix guard on `retell_llm_dynamic_variables`.
4. `packed = pack_dynamic_vars(body.retell_llm_dynamic_variables, body.metadata)`.
5. Persist `ChatSession(agent_profile_id=profile_id, agent_version=<profile.published_version>, status=ONGOING,
   chat_type="api_chat", dynamic_vars=packed, started_at=now)` (do **not** set `organization_id` — §4 note).
   **No LLM call.** `commit`.

> `create_chat` is **not** idempotent — each call mints a distinct `chat_id` (the oracle `create-chat`
> request has no `idempotency_key`, matching RetellAI).

### 8.2 `get_chat(db, chat_id) -> ChatSession`

Load by `ids.decode_chat_id(chat_id)` with `WHERE archived_at IS NULL`; `CompatError(404)` if missing or
archived. Serialized by the router with `include_transcript=True` (200, `ChatResponse`). The org-scoped
session means a `chat_id` minted under another org's key resolves to no row → 404 (cross-tenant isolation).

### 8.3 `create_chat_completion(db, settings, body) -> list[ChatMessage]` (the live path)

Order is deliberate — **no PHI is written before validation, and the session row is locked first**:

1. Load the session by `ids.decode_chat_id(body.chat_id)` **`… FOR UPDATE`** (`WHERE archived_at IS NULL`);
   `CompatError(404)` if missing/archived. The row lock serializes concurrent completions and makes the
   `seq` computation (§4.3) safe.
2. `CompatError(422, "chat is not ongoing")` if `session.status != ChatStatus.ONGOING`.
3. **`if not settings.gcp_project: raise CompatError(503, "chat completion unavailable")`** — *before any
   write* (mirrors `admin_profile_tests.py:133`). Nothing is persisted when Vertex is unconfigured.
4. Persist incoming `ChatMessage(role="user", content=body.content, seq=<MAX(seq)+1>)`.
5. Load the agent profile config; `(bare_vars, _) = unpack_dynamic_vars(session.dynamic_vars)`;
   `values = build_vars({}, bare_vars, timezone="", now=datetime.now(UTC))`;
   `system_instruction = substitute(cfg.prompts.system_prompt, values)`
   (mirrors the admin-test handler: bare user vars go in the `custom` slot, `timezone=""`).
6. Build `contents` from the full prior history **plus** the new user turn, as
   `{"role": "model" if m.role == "agent" else "user", "parts": [{"text": m.content}]}`
   (i.e. `agent → model`, `user → user`; the exact shape `admin_profile_tests.py` uses).
7. `turn = await run_vertex_turn(model=cfg.llm.model, temperature=cfg.llm.temperature,
   system_instruction=system_instruction, tools=[], contents=contents, settings=settings)`.
8. Persist `ChatMessage(role="agent", content=turn.text, seq=<MAX(seq)+1>)`; `commit`.
9. Return **only the new agent message(s)** (excludes the input — §3.2).

**Failure modes:**

- `settings.gcp_project` unset → 503 at step 3 (nothing written; see above).
- `run_vertex_turn` raises → `await db.rollback()` (the whole uncommitted transaction — the user-turn insert
  is discarded, no partial PHI exchange persists), then `CompatError(502, "chat completion failed") from None`
  (global handler logs `type(exc).__name__`). The session stays `ONGOING` (client may retry); `ChatStatus.ERROR`
  is **not** assigned in 4a.

### 8.4 `update_chat`, `end_chat`, `delete_chat`

- `update_chat`: load (`archived_at IS NULL`, 404 otherwise); `(old_vars, old_meta) =
  unpack_dynamic_vars(session.dynamic_vars)`; if `override_dynamic_variables` provided → replace bare vars
  (reserved-prefix guard); if `metadata` provided → replace metadata; persist `custom_attributes` if provided;
  `session.dynamic_vars = pack_dynamic_vars(new_vars, new_meta)` (no `__meta_unhonored__` in 4a);
  `data_storage_setting` accepted-and-ignored; `commit`; return serialized (200, `ChatResponse`).
- `end_chat`: load (404 if missing/archived); `status=ENDED`, `ended_at=now`; `commit`; 204. (An already-ended
  chat is still GET-able; only `create-chat-completion` to it 422s.)
- `delete_chat`: load (404 if already archived/missing); `archived_at=now` (soft-delete; audit-preserving);
  `commit`; 204. Subsequent get/list exclude it (404 on get).

### 8.5 `list_chats(db, body) -> (items, pagination_key, has_more, total?)`

Mirror the existing compat `list-calls` implementation: `WHERE archived_at IS NULL`, cursor by
`(start_timestamp, id)`, `limit`+1 probe for `has_more`, opaque `pagination_key`, `skip`, `include_total`.
Serialize each item with **`include_transcript=False`** (§7) → **no per-item `chat_messages` query** (the
list path loads only `chat_sessions` rows, avoiding the list-calls N+1 that was a pre-GA blocker in earlier
phases). **Honored** filters: `filter_criteria.agent` (by `agent_id`) and `chat_status`.
**Accepted-not-honored** (validated for shape, not applied): `user_sentiment`, `chat_successful`,
cost/duration ranges, custom-field filters — documented in the deployment note and §12 (no post-chat
analysis exists in 4a to filter on).

## 11. Security & PHI

- **RLS:** both new tables `TenantScoped` + `FORCE RLS`; every route runs under the org-scoped compat session
  (`get_compat_db` → `set_tenant_context` + `after_begin` re-apply). `organization_id` is DB-default-filled
  from the session (§4 note) — never set by app code, never NULL. A `chat_id` minted under org A resolves to
  no row under org B's key (404).
- **PHI-safe logging:** chat content + dynamic vars are PHI. Audit logs carry only org id, op, chat id;
  exception detail is logged as `type(exc).__name__` only by the global handler; services log no PHI (§8).
  Vertex egress is BAA-covered (Vertex AI, ADC; never the Developer API).
- **No bearer token concept** in chat (unlike web calls) — `ChatResponse` (§3.1) and all request bodies (§3.5)
  contain no token/auth field, so there is nothing token-shaped to leak. Auth is enforced only at the
  router/session layer (`get_compat_db`).
- **Reserved-prefix guard** on every var-write path (`create-chat`, `update-chat`); the reserved
  `__meta_unhonored__` key is stripped on `unpack_dynamic_vars` (defense-in-depth) though 4a persists nothing
  under it.
- **`extra="forbid"`** on request models rejects unknown fields with a 422 envelope.

## 12. Deviations from the oracle (documented in the deployment note)

1. `chat_type` is always `api_chat` in 4a (`sms_chat` → 4b).
2. `chat_analysis`, `collected_dynamic_variables`, `chat_cost`, `custom_attributes` (in responses) are omitted
   (no post-chat analysis yet); `rerun-chat-analysis` stays 501.
3. Agent replies are `role=agent` **text only** — no `tool_call_invocation`/`tool_call_result`/transition/
   injected/sms message roles emitted.
4. `/v3/list-chats` rich filters (`user_sentiment`, `chat_successful`, cost/duration ranges, custom fields)
   are accepted-but-not-honored.
5. `update-chat`'s `data_storage_setting` is accepted-and-ignored (not persisted; 4a has no data-storage controls).
6. `create-chat-completion` is synchronous (one Vertex turn per call); RetellAI may stream — our reply is a
   single completed agent message, which is shape-conformant.
